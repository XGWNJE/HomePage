## The failure happened during Codex visual verification

The incident occurred while Codex Desktop was visually verifying the local blog through the user's real Chrome profile. The page appeared to open correctly: Codex could read its title and URL, and a complex page such as Bilibili captured normally. Only local blog HTML stalled when Codex requested a screenshot, DOM snapshot, or raw Chrome debugging command. The JavaScript control session then reset, making the failure look like “the Codex Chrome plugin crashes on every screenshot.”

Chrome itself did not exit, and the ChatGPT Chrome Extension remained installed and connected. The session reset was the consequence of a timeout: the page renderer could not answer Codex's visual inspection commands.

> [!NOTE]
> This was not a universal Codex screenshot failure. It affected only HTML pages that loaded this site's shared frontend runtime. Remote pages, local static images, and image-byte transport all worked.

## Split the Codex visual verification pipeline

The Codex-to-Chrome screenshot path has at least three stages: generate image bytes in Chrome, return them through the extension and native bridge, and display them in Codex. Combining all three into one call only produces an ambiguous timeout.

The diagnosis therefore tested them separately: capture without displaying, emit already-generated bytes, and compare blank, remote, local-static, and local-HTML pages.

Blank pages and Bilibili generated screenshots successfully, and hundreds of kilobytes were returned in milliseconds. Image size, transport, and a globally broken Codex Chrome connection were no longer credible explanations.

## Astro, MDX, and the dev server were not the cause

The same page was served from a production build preview instead of the development server. If HMR, WebSockets, or MDX compilation were responsible, the static preview should have recovered. It did not.

A 32×32 PNG on the same origin captured in 83 milliseconds. The boundary was now clear: only HTML pages that started the shared site runtime became unresponsive.

## A single-variable test isolated the cursor

The site enables its custom cursor only for a fine pointer. A mouse is normally fine; a touchscreen is usually coarse.

Emulating reduced motion made the screenshot complete in 91 milliseconds, placing the bug somewhere in the motion-related path. A second experiment kept ordinary animations but emulated a coarse pointer. Capture still recovered, in 175 milliseconds. The important branch skipped by that experiment was the custom cursor.

## Root cause: an observer watched its own writes

To keep the native cursor hidden across route changes, the cursor runtime observed `class` and `style` attributes on `html` and `body`. Its callback then wrote those same attributes unconditionally:

```ts
const suppressNativeCursor = () => {
  document.documentElement.classList.add('has-custom-cursor');
  document.documentElement.style.cursor = 'none';
  if (document.body) document.body.style.cursor = 'none';
};
```

That created a feedback loop: an attribute changed, the observer ran, the callback wrote the attribute, and the observer ran again. The first frame could already be visible, so the page looked loaded. Renderer-dependent operations such as screenshots, DOM snapshots, and debugging commands still starved and timed out.

## The fix: make restoration genuinely idempotent

The custom cursor and animations were not removed. The fix only checks whether a change is necessary before writing:

```ts
const suppressNativeCursor = (): void => {
  const root = document.documentElement;
  if (!root.classList.contains('has-custom-cursor')) root.classList.add('has-custom-cursor');
  if (root.style.cursor !== 'none') root.style.cursor = 'none';
  if (document.body && document.body.style.cursor !== 'none') document.body.style.cursor = 'none';
};
```

The observer can still restore state after external changes. The next callback caused by that restoration sees that the values are already correct and performs no writes, so the loop ends.

Post-fix verification was repeated by Codex with a real fine pointer and reduced motion disabled. Exactly one custom cursor existed, the native cursor remained hidden, `tab.screenshot()` completed, and all article components rendered normally.

## Why this matters to other Codex users

Codex visual verification is more than opening a URL. The browser extension, Native Messaging bridge, tab debugging session, and page renderer must all cooperate. A readable title, a correct URL, and even a visible first frame do not prove that screenshots or DOM inspection can complete.

For the specific “navigation works, screenshots do not” pattern, a useful order of operations is:

1. Capture a complex remote page to check whether Chrome control is globally healthy.
2. Separate screenshot generation from image transport.
3. Compare a static resource and a real HTML page on the same localhost origin.
4. Test screenshot, DOM snapshot, and CDP paths; if all three stall, investigate the renderer or debugging session before the image format.
5. Distinguish a timed-out control-session reset from an actual Chrome process crash.

## The engineering habit worth keeping

> Split the pipeline, then change one variable at a time. The most visible failing action is not automatically the root cause.

The durable lesson is a repeatable diagnostic sequence:

1. Separate capture, transport, and presentation stages.
2. Build blank, remote, local-static, and local-dynamic control cases.
3. Change one environmental variable per experiment and record comparable results.
4. Use runtime evidence to narrow the search before reading source code.
5. Verify that the original feature still works after the error disappears.

Codex did not mysteriously break Chrome by taking a screenshot. A page-level visual-consistency safeguard accidentally formed a self-reinforcing loop. Once Codex, the extension, transport, and rendering were separated into measurable boundaries, the bug became ordinary—and fixable.
