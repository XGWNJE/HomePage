# README 编辑部式索引视觉改造设计

日期：2026-07-18

## 目标

把仓库首页从“信息完整的维护文档”重构为“先建立项目认知、再提供真实证据、最后引导行动”的 GitHub 项目首页，同时替换已经不能代表当前产品的桌面端与移动端截图。

README 需要让首次访问者在首屏回答三个问题：

1. XGWNJE 是什么：把研究、工程与长期笔记组织成可持续发布、检索和阅读的个人索引。
2. 它现在是什么样：直接展示当前线上主页与文章移动端的真实截图。
3. 下一步做什么：访问站点，或用最短命令在本地启动。

## 受众与核心叙事

- 主要受众：从 GitHub 进入项目的读者、潜在协作者和未来维护者。
- 一句话价值：把分散的内容与工程能力收拢成一套长期可维护的个人索引。
- 主要证据：当前生产站的桌面主页与移动端文章截图。
- 首个成功动作：打开线上主页，或执行 `npm ci` 与 `npm run dev`。
- 叙事顺序：价值 → 真实产品 → 工作方式 → 快速开始 → 技术边界与文档地图。

## 视觉方向

采用“编辑部式索引”方向，延续站点现有的暗色、古典、档案与终端语义，不制作通用 SaaS 风格封面。

### 视觉系统

- 色彩：近黑背景、暖白正文、旧金强调色、低饱和玫红状态色、灰褐辅助文字。
- 字体：系统衬线字体用于项目名与主题句，系统无衬线用于说明，等宽字体用于元数据和流程标签；不加载远程字体。
- 形状：细描边、克制圆角、索引卡片、纵向栏线与终端光标。
- 主题图形：把 NOTE / CODE / IMAGE 汇入 ASTRO INDEX，再输出到 WEB / RSS / API，作为项目工作方式的视觉摘要。
- 构图：左侧负责身份与价值，右侧负责机制；背景使用轻量网格和古典栏线，不使用无意义装饰。

## README 信息架构

1. 项目原生 SVG 主视觉。
2. 居中的线上入口与关键链接。
3. 当前生产主页大图，作为第一证据。
4. 简短的“这是什么”和三项核心能力。
5. 最短可运行路径与普通文章发布入口。
6. 当前架构图与边界说明。
7. 移动端文章证据。
8. 仓库地图、维护文档、上游与许可证。

README 保留必要的技术事实，但删除首屏中的发布流程细节和重复解释。完整发布边界继续由维护文档负责。

## 素材方案

新增并由 README 实际引用：

- `assets/readme/hero.svg`：可维护的项目原生主视觉。
- `assets/readme/home-desktop.webp`：当前生产主页桌面截图。
- `assets/readme/article-mobile.webp`：当前生产文章移动端截图。

删除被替换且无其他引用的旧素材：

- `docs/assets/home-desktop.webp`
- `docs/assets/article-mobile.webp`

截图只做裁切、缩放与 WebP 压缩，不修改产品内容。SVG 使用纯矢量、系统字体、内嵌背景、`title` 与 `desc`，不使用脚本、远程字体或 `foreignObject`。

## 实施边界

- 只修改 README、README 专用素材与本设计/计划文档。
- 不改站点前端、后端、内容模型、部署配置或生产环境。
- 不增加运行时依赖。
- 不提交构建产物、临时预览图或浏览器 profile。
- 本轮不主动提交、推送、创建 PR 或发布。

## 验证标准

- README 首屏能明确表达产品定位、真实产品状态与下一步行动。
- 所有本地图片路径存在，旧截图不再被引用。
- SVG 在浅色与深色 GitHub 页面上都具有自己的完整背景和可读对比度。
- 桌面端与窄屏下，关键 Markdown 文本不依赖图片仍可理解。
- README 通过 `beautify-github-readme` 自带审计脚本。
- SVG 可被标准 XML 解析；截图为有效 WebP；文本保持 UTF-8、无 BOM、无替换字符。
- `git diff --check` 通过，并只留下预期范围内的变更。
